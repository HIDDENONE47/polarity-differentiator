# Methodology

**PolarityIQ — Stage 1 Differentiator Assessment: Family Office Micro-RAG**

This document describes how the dataset was discovered, extracted, validated, and made queryable — with reference to the actual functions and modules in this repository.

---

## 1. Data Discovery & Enrichment

Discovery and extraction are handled by two cooperating modules: `graph_orchestrator.py` (control flow) and `agent_extraction.py` (search + structuring logic), wired together as a LangGraph `StateGraph` with five nodes: `discover → extract → audit → filter → export`.

**Discovery.** `graph_orchestrator.py` holds a fixed list of twelve `DISCOVERY_QUERIES` — Tavily search strings deliberately varied across four axes: broad footprint queries (e.g. entities with an "investment mandate" and "AUM" language, but excluding directory/listicle pages), platform-specific queries scoped to `site:linkedin.com/company` and `site:crunchbase.com`, sector/geography-targeted queries (healthcare, real estate, fintech; Texas, California, London), and transactional queries looking for recent fund launches or direct-investment press releases. The `discover_node` pulls candidates from these queries one at a time via `rate_limited_search()`, advancing to the next query only once the current one is exhausted, and paces each call by 1 second to stay inside Tavily's free-tier request budget.

**Extraction.** For each candidate, `extract_node` calls `_fetch_full_text_for_extraction()` to pull the full page content (via Tavily's `extract` endpoint, not just the short search snippet) and hands it to `raw_extract_entity_from_source()` in `agent_extraction.py`. This function does two things in a single LLM call, using the extraction-tier Groq model (`DEFAULT_CONFIG.llm_model`) at `temperature=0`:

- **Classification first.** The prompt instructs the model to judge whether the entity is *actually* a private family office — explicitly passing multi-family offices serving a normal-sized 2–30 family client base, and explicitly rejecting institutional asset managers/OCIOs, regulatory documents, and vendors or recruiters who merely serve or mention family offices. If the model returns `is_family_office: false`, `raw_extract_entity_from_source` raises a `ValueError` with the model's stated reasoning, which `extract_node` catches and logs to `rejected_log` with a `extraction_failed:` reason rather than letting a bad candidate silently proceed.
- **Structured extraction second.** If classified as a genuine family office, the same call extracts the entity's proper name, investing thesis, mandate, background, AUM, corporate LinkedIn, entity type, location, website, named principals (name/title/LinkedIn/email/phone), and any recent signals (investments, hires, news) with dates.

The prompt enforces strict JSON-only output with no markdown fencing or prose, and the parsing layer defends against three concrete LLM formatting quirks observed in practice: markdown-wrapped JSON (stripped via regex), an outer JSON array instead of a single object (unwrapped if the first element is a dict), and stringified booleans like `"false"` (coerced explicitly via `_coerce_bool`, since Python's `bool("false")` evaluates to `True` and would otherwise let a rejected entity through silently).

Every field produced at this stage is wrapped in a `VerifiableField` with `status=UNVERIFIED` and `extraction_method=LLM_INFERENCE` — extraction is architecturally prevented from marking anything as verified itself; that authority belongs entirely to the audit layer described below.

`route_after_extract` sends successfully classified candidates on to `audit_node`; a rejected or malformed extraction routes back to `discover` to try the next candidate. The whole loop is bounded by `MAX_CANDIDATES_SAFETY_CAP = 500` and a `target_count` of 50 accepted records, and `run_pipeline()` recovers previously accepted records from `family_offices_dataset_BACKUP.jsonl` on startup, so a killed or rate-limited run resumes rather than restarting from zero.

## 2. Output Validation (Epistemic Rigor)

Validation is isolated entirely in `agent_auditor.py`, which is the sole module authorized to promote a `VerifiableField` from `UNVERIFIED` to `VERIFIED` or `COULD_NOT_VERIFY`. This separation of concerns is deliberate: the module's docstring states its reason for existing is to prevent `agent_extraction.py` hallucinating a plausible-sounding fact and having that fact silently become "verified" with no independent check.

**Single-source audit (`audit_field`).** For every wrapped field, `audit_field()` re-fetches the field's claimed `source_url` *live* via `_fetch_live_page_text()` (a Tavily `extract` call, cached per-URL in `_page_cache` since multiple fields from one record typically share a source page — this avoids re-fetching the same page five-plus times per record). The re-fetched text and the claimed value are then passed to `_adversarial_confirm()`, which prompts the auditor-tier Groq model (`DEFAULT_CONFIG.auditor_llm_model`) with explicit skeptic instructions: *"Your job is to find reasons a claim is WRONG, not reasons it might be right... Vague relatedness is not support... Silence on the fact means NOT supported."* The model returns a `{supported, reasoning, confidence}` verdict; a field is only promoted to `VERIFIED` if `supported=True` **and** `confidence >= 0.6`. Anything else — a dead link, a non-corroborating page, an unparseable auditor response, or a low-confidence verdict — resolves to `COULD_NOT_VERIFY`, never a silent default to trusting the original claim.

**Cross-source corroboration (`audit_record_deep`).** This is the strongest validation path in the system, reserved for the three spotlight records (invoked from `run_spotlight.py`) because it roughly doubles API cost versus the standard path and isn't affordable across all 50 records on the free-tier budget. `deep_audit_field()` first runs the normal `audit_field()` single-source check; only if that already passes does it call `_find_corroborating_source()`, which searches Tavily (`search_depth="advanced"`) for a *second, independent* source for the same claim, explicitly excluding the original `source_url` from the results. If a corroborating page is found, it's run through the same `_adversarial_confirm()` adjudication. The outcome is always recorded honestly in `verification_notes` rather than silently upgraded or hidden:
- Corroborated → notes append `"Cross-confirmed by independent source: <url> (<reasoning>)"`, and `confidence_score` is updated to the max of the two passes.
- No second source found → notes append `"(Single source only — no independent corroboration found.)"`.
- Second source found but disagrees → notes append `"Note: second source (<url>) did not corroborate this claim."`

`audit_record_deep()` applies this deep path to the five entity-level fields (investing thesis, mandate, background, AUM, corporate LinkedIn) — chosen because these are the fields with genuine independent secondary sources (press coverage, filings). Principal contact fields (LinkedIn, phone) still route through the standard `audit_field()`/`audit_work_email()` — a second free public source for a personal direct phone number essentially doesn't exist, so the extra API cost there wouldn't buy anything.

**Independent email verification (`audit_work_email`).** Applied to all 50 records, not just the spotlight three. Rather than re-confirming the extraction agent's claimed email against a web page, this function queries Hunter.io's Email Finder independently by domain and name. If Hunter finds nothing above a 50-confidence threshold, the field resolves to `COULD_NOT_VERIFY`. If Hunter's result *disagrees* with what extraction claimed, the field is explicitly flagged in `verification_notes` as a conflict for manual review — never silently resolved by picking one source over the other.

**Confidence scoring.** Every `VerifiableField` carries an optional `confidence_score` (0.0–1.0), set exclusively by the auditor — never by the extraction agent, which has no mechanism to self-assess confidence. This score comes directly from the adversarial LLM's own stated confidence in `_adversarial_confirm()`, or from Hunter.io's returned score (normalized to 0–1) for email fields.

**Honest blanks.** The `VerificationStatus` enum is intentionally three-valued: `VERIFIED`, `COULD_NOT_VERIFY`, and `UNVERIFIED`. The first two are the only states permitted to leave the pipeline — `is_exportable()` on `VerifiableField` explicitly excludes `UNVERIFIED`. `COULD_NOT_VERIFY` is treated as a legitimate, honest outcome — per the docstring, "an honest blank — allowed and scored as candor" — rather than something to keep re-guessing at. This is enforced structurally, not just by convention: `VerifiableField`'s Pydantic `model_validator` raises at construction time if a field is marked `VERIFIED` without both a `source_url` and `extraction_method`, or marked `COULD_NOT_VERIFY` while still holding a value. A record only makes the final 50 if it additionally clears `passes_actionability_bar()` — at least one principal with a `VERIFIED` email, phone, or LinkedIn — since, per the assessment's own framing, a record with zero usable contact paths is treated as a hole in the product, not a formatting choice.

## 3. Micro-RAG Architecture

**Stack.** FastAPI (`backend/main.py`) for the HTTP layer, `sentence-transformers` with `all-MiniLM-L6-v2` for embeddings, FAISS (`faiss-cpu`) for vector search, and Groq (`llama-3.1-8b-instant`, via `RAG_LLM_MODEL`) for grounded synthesis at query time. This LLM is deliberately a smaller/faster model than the one used in extraction or audit, since query-time latency during a live demo matters more here than in the batch pipeline.

**Chunking strategy.** `build_index.py` chunks *per field*, not per record — the docstring's stated rationale is that a query about AUM should retrieve a precise, single-fact AUM chunk rather than a noisy blob containing an entity's entire profile. `build_chunks_from_record()` flattens each record's nested Pydantic-derived JSON into independent chunks:
- The five wrapped entity-level fields (`investing_thesis`, `investing_mandate`, `background_info`, `aum`, `corporate_linkedin`) each become one chunk via `_make_chunk()`, formatted as `"{entity_name} — {field label}: {value}"`.
- Plain structural fields (`entity_type`, `location`, `website`) are also surfaced as their own chunks, tagged `status="structural"` since they never went through the `VerifiableField` wrapper.
- Each principal generates a title chunk plus up to three contact chunks (`linkedin_url`, `work_email`, `direct_phone`), scoped under an entity name of `"{entity_name} principal {principal_name}"` so retrieval can distinguish which person a contact chunk belongs to.
- Each signal becomes its own chunk if its `source` field cleared verification.

`_make_chunk()` enforces a defensive filter that only fields with `status in ("verified", "could_not_verify")` are ever embedded — an `UNVERIFIED` field is refused even as a backstop, on the assumption that the auditor should never let that state reach the exported JSONL in the first place, but the index-builder doesn't trust that assumption blindly. Every chunk retains full metadata back to its origin — `entity_name`, `field_label`, `value`, `status`, `source_url`, `verification_notes`, `confidence_score` — stored alongside the vector index in `chunk_metadata.pkl`, which is what makes citation possible at query time.

**Indexing.** Chunk text is embedded with `SentenceTransformer("all-MiniLM-L6-v2")`, L2-normalized, and stored in a `faiss.IndexFlatIP` — inner product over normalized vectors, which is mathematically equivalent to cosine similarity. This is an exact (non-approximate) flat index, an appropriate choice at this dataset's scale where exhaustive search is cheap and there's no need to trade recall for index-build speed.

**Retrieval.** `rag_retriever.py`'s `retrieve()` supports two modes. With no filters, it's a plain semantic FAISS search (`index.search`) over the whole index. With an `entity_filter` and/or `field_filter` supplied, it restricts candidates to matching chunks *first* (structured filtering) via `index.reconstruct()`, computes cosine similarity against just that subset, then ranks — structured filtering is applied before semantic ranking, not as a post-filter on top-K results, so a narrow filter can't be starved out by unrelated high-scoring chunks elsewhere in the index. Every result is subject to `MIN_SIMILARITY = 0.25`; anything below that floor is dropped regardless of rank.

**Grounded synthesis.** `answer_query()` is the full retrieve-then-generate path. If `retrieve()` returns zero chunks, the LLM is never called at all — the API returns `grounded: False` with an explicit "no verified information in the dataset matches this query closely enough to answer from" message. When chunks are available, `_build_grounded_prompt()` numbers each retrieved chunk (entity, field, value, confidence, source) and instructs the model to answer *only* from those excerpts, to state plainly what is and isn't covered rather than pad gaps with a guess, and to cite the excerpt number(s) backing every stated fact (e.g. `"...AUM of $500M [1]."`). The response is returned alongside a `sources` array (entity, field, source URL, similarity score) and a `grounded: True` flag — this is what the frontend's source ledger renders directly.

**API surface.** `main.py` exposes `/query` (the retrieval+synthesis path above), `/health` (reports whether the FAISS index actually loaded, not just process liveness), and `/stats` (chunk and unique-entity counts). The index is loaded once at startup via `_ensure_loaded()` rather than lazily on first request, so a missing index fails loudly in the startup logs instead of surfacing as an opaque 500 mid-demo.

## 4. Retrospective

**What works well.** The strict separation between extraction (`agent_extraction.py`, produces only `UNVERIFIED` claims) and audit (`agent_auditor.py`, the sole authority to promote a claim) turned out to be the right architectural decision — it means a change to the extraction prompt can never accidentally weaken what counts as "verified," because verification logic lives entirely somewhere else and doesn't trust extraction's self-report. The `VerifiableField` schema's constructor-level validation (rejecting a `VERIFIED` status with no source, or a `COULD_NOT_VERIFY` status with a lingering value) caught several would-be data-integrity bugs at write time rather than downstream at export or query time. Field-level chunking for the RAG layer also paid off directly: because each chunk is one fact, `/query` responses cite a specific field and source rather than a whole-entity paragraph, which is what makes the grounding claim actually checkable rather than just asserted.

**Limitations encountered.**
- **Rate limits.** Groq's free tier enforces per-model daily token caps, which surfaces as `RateLimitError` (429) under any real batch load. `invoke_llm_with_retry()` (`config.py`) handles this with exponential backoff (starting at 10s, doubling per attempt, 6 attempts before failing loud), and separating the extraction model from the smaller auditor model gives audit its own quota pool instead of the two competing for the same 70B-model budget. Tavily's free-tier request budget is a second constraint — `rate_limited_search()` and the per-field `time.sleep(rate_limit_delay)` pacing in `audit_record()` exist specifically to stay under it across a 50-record run, and `_page_cache` in the auditor avoids redundant re-fetches of a shared source page.
- **LLM formatting quirks.** Smaller/faster models on Groq did not reliably return clean JSON on the first pass — observed failure modes included markdown code-fence wrapping, an outer array instead of a single object, and stringified booleans (`"false"` as a string, which Python's `bool()` would misread as truthy). All three are handled defensively in `agent_extraction.py` and `agent_auditor.py` via regex extraction of the JSON block and explicit type coercion, rather than assuming the model's output can be trusted as-is.
- **Cost-bounded corroboration.** True cross-source corroboration (`audit_record_deep`) roughly doubles API cost per field, which is why it's scoped to 3 spotlight records rather than applied to the full 50 — a real trade-off between epistemic confidence and free-tier budget, not a technical limitation.
- **Flat CSV export.** `_flatten_record_for_csv()` only promotes the *first* principal to top-level CSV columns for readability; the full principal and signal lists survive only as JSON strings in sidecar columns, which is fine for the RAG ingestion path (which reads the JSONL, not the CSV) but means the CSV alone under-represents entities with multiple actionable contacts.

**What I'd improve next.** Move from an exact `IndexFlatIP` FAISS index to a lightweight approximate index (e.g. HNSW) as a forward-looking scaling decision, even though it's unnecessary at 50 records — the retrieval code path wouldn't need to change, only the index type. Extend `audit_record_deep`'s corroboration path beyond the 3-record spotlight to the full dataset once cost allows, since the biggest gap between the spotlight records and the rest of the 50 is exactly that second-source check. Finally, the `entity_filter`/`field_filter` parameters in `retrieve()` are currently exact-substring matches on `entity_name`/`field_label` — replacing that with a lightweight fuzzy match would make the structured-filter path more forgiving of how a user actually phrases an entity name in a query.
